本章描述Ack/Nak协议——一种自动的、基于硬件的机制，用于确保TLP在链路上的可靠传输。接收方使用Ack DLLP来确认成功接收TLP，而Nak DLLP则指示发生了传输错误。本章描述正常操作规则以及错误恢复机制。

## 10.1 为什么需要Ack/Nak协议？

PCIe与PCI和PCI-X总线不同，它不是共享总线，而是使用点对点连接。因此，PCIe不需要像PCI那样的总线仲裁，也不会遇到PCI中由于总线仲裁引起的各种问题。然而，点对点连接引入了新的问题：如何确保数据包在链路上可靠传输？

Ack/Nak协议是PCIe解决这个问题的方案。这是一种自动的、基于硬件的机制，用于确保TLP（事务层包）在链路上的可靠传输。发送方在发送TLP后等待接收方的确认，如果收到Ack（确认），则表示TLP已成功接收；如果收到Nak（否定确认），则表示TLP传输失败，需要重传。

## 10.2 Ack/Nak协议概述

### 10.2.1 基本概念

Ack/Nak协议的核心思想很简单：

- **发送方（Transmitter）**：发送TLP的设备
- **接收方（Receiver）**：接收TLP的设备

当发送方发送TLP时，接收方会检查TLP是否有错误。如果没有错误，接收方会返回一个Ack DLLP；如果有错误，接收方会返回一个Nak DLLP。

发送方收到Ack表示接收方已成功接收至少一个TLP。收到Nak则表示接收方至少有一个TLP接收出错，发送方将重新发送相应的TLP。这是合理的，因为导致传输错误的原因通常是瞬态事件，重传很有可能解决问题。

## 10.3 发送方元素

### 10.3.1 发送方元素概述

当TLP从事务层到达时，数据链路层会执行几个操作来准备它们以进行鲁棒的错误检测：

[原文: 图10-4, 第322页]

### 10.3.2 NEXT_TRANSMIT_SEQ计数器

**NEXT_TRANSMIT_SEQ**（下一个发送序列号）计数器生成将分配给下一个传入TLP的序列号。这是一个12位计数器，在复位或当链路层报告DL_Down（链路层不活跃）时初始化为0。

由于它随每个TLP连续递增且只向前计数，计数器最终会达到最大值4095，然后回滚到0继续计数。

分配给TLP的序列号将被接收方用于在Ack或Nak中引用该TLP。虽然这个计数器很大，意味着可以有大量未确认的TLP在传输中，但实际上这不太可能发生。主要原因是接收方有要求在一定时间内为成功接收的TLP发送Ack。

[原文: 第321页]

### 10.3.3 LCRC生成器

LCRC（Link CRC，链路CRC）生成器基于要发送的头部和数据生成一个32位CRC（循环冗余校验）码，并将其添加到传出数据包的末尾以进行错误检测。

CRC使用TLP的所有字段计算，包括序列号。接收方将进行相同的计算并将其结果与TLP中的LCRC字段进行比较。如果不匹配，则在接收方的链路层检测到错误。

CRC在串行传输中很常见，因为它们：
- 在硬件中易于实现
- 擅长检测突发错误（一串错误的位）

[原文: 第321页]

### 10.3.4 重放缓冲区（Replay Buffer）

重放缓冲区（或重试缓冲区）按传输顺序存储TLP，包括序列号和LCRC。当发送方收到Ack表示TLP已成功到达接收方时，它会从重放缓冲区中清除那些序列号等于或早于Ack中数字的TLP。

由于数据包必须按顺序查看，如果收到Ack且序列号为7，则不仅TLP 7被成功接收，它之前的所有数据包也必须已被成功接收，因此没有理由在重放缓冲区中保留它们的副本。

如果收到Nak，Nak中的序列号仍然表示最后收到的良好数据包。因此，即使收到Nak，发送方也可以从重放缓冲区中清除TLP。但是，因为它是Nak，意味着接收方没有成功接收某些内容，所以在清除所有已确认的TLP后，发送方必须按顺序重放缓冲区中剩余的所有内容。

**示例**：如果收到Nak且序列号为9，则数据包9及之前的所有数据包从重放缓冲区中清除，因为接收方确认它们已成功接收。但是，因为是Nak，发送方必须然后按顺序重放缓冲区中剩余的所有TLP，从数据包10开始。

[原文: 第321-322页]

### 10.3.5 REPLAY_TIMER计数器

REPLAY_TIMER本质上是一个看门狗定时器。它确保发送方及时收到已发送TLP的Ack/Nak数据包。

**REPLAY_TIMER规则**：

| 条件 | 操作 |
|------|------|
| 如果定时器未运行 | 在任何TLP的最后一个符号传输时启动 |
| 收到表示前进的Ack，且重放缓冲区仍有未确认TLP | 定时器重置并重新启动 |
| 重放事件发生，且第一个重放TLP的最后一个符号已发送 | 定时器重置并重新启动 |
| 没有TLP要传输，或重放缓冲区为空 | 定时器重置并保持 |
| 收到Nak | 定时器重置并保持；在第一个重放TLP的最后一个符号发送时重新启动 |
| 定时器超时 | 定时器重置并保持；在第一个重放TLP的最后一个符号发送时重新启动 |
| 数据链路层不活跃 | 定时器重置并保持 |
| 链路训练或再训练期间 | 定时器保持 |

> **符号时间说明**：
> - Gen1 (2.5 GT/s): 1符号时间 = 4ns
> - Gen2 (5 GT/s): 1符号时间 = 2ns
> - Gen3 (8 GT/s): 1符号时间 = 1.25ns

[原文: 第323、337页]

### 10.3.6 REPLAY_NUM计数器

REPLAY_NUM是一个2位计数器，用于跟踪收到Nak或REPLAY_TIMER超时后的重放尝试次数。

当REPLAY_NUM计数从11b滚动到00b（表示4次尝试传递同一组TLP失败）时，数据链路层自动强制物理层重新训练链路（LTSSM进入Recovery状态）。

如果实现了可选的高级错误报告寄存器，还会设置Replay Number Rollover错误状态位。

[原文: 第323、336页]

### 10.3.7 ACKD_SEQ寄存器

ACKD_SEQ是一个12位寄存器，存储最近收到的Ack或Nak的序列号。它在复位或数据链路层不活跃时初始化为全1。

该寄存器用收到的Ack或Nak中的AckNak_Seq_Num[11:0]字段更新。ACKD_SEQ计数与最后收到的Ack或Nak中的序列号进行比较以检查前进进度。

如果最新的Ack/Nak的序列号比ACKD_SEQ寄存器中的晚，则表示正在取得前进进度。

**重要限制**：NEXT_TRANSMIT_SEQ和ACKD_SEQ计数器不允许被超过其总计数值的一半分隔。如果发送了大量未确认的TLP，使得NEXT_TRANSMIT_SEQ计数值比ACKD_SEQ计数晚2048，则在收到更多Ack解决此问题之前，将不再从事务层接受TLP。

[原文: 第323-324页]

### 10.3.8 DLLP CRC检查

此块检查DLLP中16位CRC的错误。如果检测到错误，DLLP被丢弃，如果启用，可能会报告可纠正错误。不采取进一步行动，因为没有机制来重放或纠正失败的DLLP。

相反，我们只需等待下一个成功的Ack/Nak，这将使计数器恢复最新状态并允许正常操作继续。

[原文: 第324页]

## 10.4 接收方元素

### 10.4.1 接收方元素概述

传入的TLP首先检查LCRC错误，然后检查序列号。如果没有错误，TLP被转发到接收方的事务层。如果有错误，TLP被丢弃，并且除非已经有Nak未完成，否则会调度Nak。

[原文: 图10-5, 第325页]

### 10.4.2 LCRC错误检查

此块通过验证32位LCRC来检查接收TLP中的传输错误。此块基于接收的TLP位计算LCRC值，然后将计算的LCRC与接收的LCRC进行比较。

如果匹配，则数据包的所有位都完全按照传输时接收。如果不匹配，则TLP中存在位错误，因此将其丢弃并发送Nak以重放该数据包和坏数据包之后发送的任何TLP。

[原文: 第325页]

### 10.4.3 NEXT_RCV_SEQ计数器

**NEXT_RCV_SEQ**（下一个接收序列号）计数器跟踪预期的序列号，用于验证顺序数据包接收。它在复位或数据链路层不活跃时初始化为0，并为每个转发到事务层的良好TLP递增一次。

有错误或已被取消的TLP不会发送到事务层，因此不会递增此计数器。

**复位条件**：
- 数据链路层进入DL_Down状态
- 数据链路层从DL_Down状态退出
- 链路训练完成（进入L0状态）
- 系统复位

[原文: 第326页]

### 10.4.4 序列号检查

如果LCRC检查通过，接收方接下来将NEXT_RCV_SEQ计数器与新接收TLP中应有的序列号进行比较。

**三种可能的结果**：

1. **序列号等于NRS计数（预期数字）**
   - 一切正常：接受TLP并转发到事务层
   - NRS计数递增
   - 接收方调度Ack，但不必在AckNak_LATENCY_TIMER到期之前发送
   - 与此同时，可能收到其他良好的TLP，递增NEXT_RCV_SEQ计数器
   - 一旦定时器到期，发送单个Ack，序列号为最后收到的良好TLP（NRS - 1）

2. **序列号早于NRS计数（小于预期）**
   - 此TLP之前已见过，是重复数据包
   - 只要预期序列号和接收序列号不被超过总计数值的一半（2048）分隔，这就不是错误
   - 重复数据包被静默丢弃（无Nak，无错误报告）
   - 发送Ack，序列号为最后收到的良好TLP（NRS - 1）
   - 为什么会这样？发送方可能没有收到之前发送的Ack，因此其REPLAY_TIMER超时并重传了重放缓冲区中的所有内容

3. **序列号晚于NEXT_RCV_SEQ计数（大于预期）**
   - 链路层丢失了一个TLP
   - 例如，如果预期序列号30而传入TLP有序列号31，则知道有问题
   - 数字必须是顺序的，既然不是，则一定有一个失败并在物理层被丢弃
   - 此无序TLP被丢弃，无论它是否有其他错误，因为必须按顺序接受TLP
   - 如果没有已经未完成的Nak，则发送Nak

[原文: 第326-327页]

### 10.4.5 NAK_SCHEDULED标志

此标志在接收方调度Nak时设置，在接收方成功接收具有预期序列号（NRS）的TLP时清除。

规范明确说明，当NAK_SCHEDULED标志保持设置时，接收方不得调度额外的Nak DLLP。这是为了防止可能的无限循环：发送方开始重放一些数据包，但接收方在重放完成之前发送另一个Nak，导致它重新开始发送。

[原文: 第327页]

### 10.4.6 AckNak_LATENCY_TIMER

此定时器控制接收方发送Ack DLLP的时机，允许接收方合并多个TLP的确认。

**启动条件**：
- 当收到一个良好TLP时
- NAK_SCHEDULED标志未设置（如果已设置，会立即发送Nak而不是等待定时器）

**定时器行为**：
- 定时器运行的时长由规范规定，确定接收方可以合并Ack的时间长度
- 一旦AckNak延迟定时器到期，生成并发送序列号为NRS-1的Ack，指示最后收到的良好数据包
- 此定时器在发送Ack或Nak时重置
- 只有在收到新的良好TLP时才重新启动

**目的**：通过允许接收方合并多个TLP的确认，减少链路上的Ack DLLP数量，提高带宽利用率。

[原文: 第328、343页]

### 10.4.7 Ack/Nak生成器

Ack或Nak DLLP由错误检查块调度，包含12位AckNak_Seq_Num字段。它通过从NRS计数减去1来计算此数字，结果为报告最后收到的良好序列号。

这是因为良好的TLP接收在调度Ack之前递增NRS，而失败的TLP只是调度Nak而不递增NRS。

**唯一例外**：复位后的第一个TLP。如果第一个TLP（使用序列号0）失败，结果Nak的AckNak_Seq_Num值为零减1，结果为全1。

**Ack/Nak DLLP格式**：

```
Byte 0:  DLLP类型
         0000 0000b = Ack
         0001 0000b = Nak

Byte 1:  保留

Byte 2-3: AckNak_Seq_Num[11:0] (序列号)

Byte 4-5: 16位CRC
```

[原文: 第328-329页]

## 10.5 Ack/Nak协议详细说明

### 10.5.1 发送方协议详情

**序列号分配**：
- 当事务层将TLP交付给链路层时，首先附加一个12位序列号
- 下一个增量序列号实际上可能更小，如计数器在达到最大值4095后回滚到零时发生的那样

**32位LCRC**：
- 发送方基于TLP内容（序列号、头部、数据有效载荷和ECRC）生成并附加32位LCRC

**重放缓冲区**：
- 在设备传输TLP之前，它将TLP的副本存储在重放缓冲区中
- 每个缓冲区条目存储完整的TLP及其所有字段：序列号（12位宽，占用两个字节）、头部（最多16字节）、可选数据有效载荷（最多4KB）、可选ECRC（4字节）和LCRC字段（4字节）

**重放缓冲区大小**：
- 规范选择不指定重放缓冲区大小，留给设备设计人员优化
- 应该足够大以存储尚未被Ack确认的TLP，以便在正常操作条件下不会变满并阻塞来自事务层的新TLP
- 设计人员考虑的因素：
  - 来自接收方的Ack DLLP延迟
  - 物理链路引起的延迟
  - 接收方从L0s到L0的退出延迟

[原文: 第329-330页]

### 10.5.2 发送方对Ack DLLP的响应

接收方返回的单个Ack可以确认多个TLP；不是每个传输的TLP都需要专用Ack。

接收方可以接收多个良好的TLP并发送一个Ack，序列号为最后收到的良好TLP。发送方对取得前进进度（序列号比上次看到的晚）的Ack的响应是：
- 用新Ack的序列号加载AckD_SEQ寄存器
- 重置REPLAY_NUM计数器和REPLAY_TIMER
- 清除被该Ack确认的所有TLP的重放缓冲区

[原文: 第330-331页]

### 10.5.3 发送方对Nak的响应

Nak表示发生了问题。当发送方收到Nak时，它首先从重放缓冲区中清除任何序列号较早或相等的TLP，然后从Nak中序列号之后的序列号开始重放剩余的TLP。

如果Nak导致至少一个TLP从缓冲区中清除，则我们取得了前进进度。在这种情况下，发送方重置REPLAY_NUM计数器和REPLAY_TIMER，并用Nak的序列号加载AckD_SEQ寄存器。

**TLP重放**：
- 当需要重放时，发送方阻塞从事务层接受新TLP
- 然后按缓冲区中放置的顺序（如FIFO）重放必要的TLP
- 重放事件后，数据链路层解除对从事务层接受新TLP的阻塞。重放的TLP保留在缓冲区中，直到最终在稍后的某个时间被确认。

**高效TLP重放**：
重放期间收到的Ack或Nak DLLP必须被处理。主要有两个选项：
1. 发送方可以将它们保留到重放完成，然后评估Ack或Nak并采取适当步骤
2. 另一个选项是在重放期间开始处理新收到的Ack/Nak DLLP

如果使用第二个选项，新收到的Ack可能会在重放进行过程中清除缓冲区中的一些条目，可能减少需要重放的TLP数量并节省链路上的时间。这是允许的，但重要的是要记住，一旦TLP开始传输，就必须完成它。

[原文: 第333-334页]

### 10.5.4 接收方协议详情

**物理层**：
在物理层接收的TLP被检查接收错误（如帧、 disparity和无效符号）。如果在此级别有错误，TLP被丢弃，链路层可能通过某种设计特定的方法被告知，以便它可以调度Nak并让数据包重放。如果链路层未被告知，则最终它会检测到序列号违规，这将导致Nak和重放。

**TLP LCRC检查**：
如果没有物理层错误，链路层首先检查CRC错误。接收方从接收的TLP（不包括LCRC字段）计算预期的LCRC值，并将此值与TLP的32位LCRC进行比较。

如果两个匹配，则TLP良好。否则，TLP被丢弃，接收方调度Nak。

**下一个接收TLP的序列号**：
如果LCRC正确，接收方接下来将NEXT_RCV_SEQ计数器与新接收TLP中应有的序列号进行比较。

在正常操作条件下，这两个数字将匹配。如果匹配，接收方将TLP转发到事务层，递增NEXT_RCV_SEQ计数器，并调度Ack。

**重复TLP**：
如果传入数据包的序列号早于预期值，这意味着发送方决定重新发送接收方之前已经见过的数据包。此重复数据包不是错误，尽管我们在链路上浪费时间重新发送它。

这可能是由于发送方的超时引起的，如果之前TLP的Ack或Nak失败。当在接收方看到这种情况时，重复数据包被丢弃，并调度Ack，序列号为最后收到的良好TLP（可能与重放TLP中的序列号不同）。

**无序TLP**：
如果传入数据包的序列号晚于预期值，唯一的解释是一定丢失了一个TLP。这是可纠正的错误，通过发送Nak来处理。

无论传入数据包是否良好都无关紧要，因为它们只能按正确的序列号顺序接受。数据包被丢弃，接收方等待具有预期序列号的TLP。

当TLP收到CRC错误、被取消或序列号检查失败时，NEXT_RCV_SEQ计数器不会递增。

[原文: 第340-342页]

## 10.6 Ack/Nak示例

### 10.6.1 示例1：Ack示例

**说明**：
1. Device A发送序列号为3、4、5、6、7的TLP
2. Device B成功接收TLP 3，递增NEXT_RCV_SEQ计数器从3到4，启动AckNak_LATENCY_TIMER
3. Device B在定时器到期前成功接收TLP 4和5
4. 定时器到期后，Device B发送Ack，序列号为5，确认收到TLP 5及之前的所有TLP
5. Device A收到Ack 5，重置REPLAY_TIMER和REPLAY_NUM计数器，清除TLP 3-5

[原文: 第331-332页]

### 10.6.2 示例2：序列号回滚的Ack

此示例展示序列号回滚行为：

1. Device A发送序列号为4094、4095、0、1、2的TLP（TLP 4094是第一个发送的，TLP 2是最后一个）
2. Device B按顺序成功接收TLP 4094、4095、0、1。接收TLP 4094启动AckNak_LATENCY_TIMER。TLP 4095、0和1在定时器到期前被接收。TLP 2仍在传输中。
3. 定时器到期，Device B发送Ack，序列号为1，确认收到TLP 1及之前的所有TLP（本例中为0、4095和4094）
4. Device A成功收到Ack 1，从重放缓冲区中清除TLP 4094、4095、0和1，重置REPLAY_TIMER和REPLAY_NUM计数器

[原文: 第332页]

### 10.6.3 示例3：Nak示例

**说明**：
1. Device A发送序列号为4094、4095、0、1、2的TLP
2. Device B成功接收TLP 4094，NEXT_RCV_SEQ计数器递增到4095，启动AckNak_LATENCY_TIMER
3. Device B在下一个接收的TLP（TLP 4095）中检测到CRC错误，设置NAK_SCHEDULED标志，发送Nak，序列号为4094（NEXT_RCV_SEQ计数 - 1）
4. Device B不会等待AckNak_LATENCY_TIMER到期再发送Nak，通常在下一个数据包边界发送
5. 因为Nak已调度传输，AckNak_LATENCY_TIMER停止并重置
6. Device A收到Nak 4094，从重放缓冲区中清除TLP 4094及之前的TLP（本例中无），然后重放缓冲区中剩余的所有TLP（TLP 4095、0、1和2）

[原文: 第334-335页]

### 10.6.4 示例4：丢失TLP的处理

**说明**：
1. Device A发送TLP 4094、4095、0、1、2
2. Device B成功接收TLP 4094，启动AckNak_LATENCY_TIMER，递增NEXT_RCV_SEQ计数器。之后还接收TLP 4095和0
3. 收到TLP 0后，AckNak_LATENCY_TIMER到期，导致调度Ack，序列号为0
4. 看到Ack 0，Device A从重放缓冲区中清除TLP 4094、4095和0
5. TLP 1在途中丢失（可能物理层丢弃了它），TLP 2到达。序列号检查显示TLP 2的序列号不等于NEXT_RCV_SEQ计数，而是在无序范围内
6. Device B丢弃TLP 2并设置NAK_SCHEDULED标志，发送Nak 0（NEXT_RCV_SEQ计数 - 1）
7. 收到Nak 0后，Device A重放TLP 1和2

[原文: 第345-346页]

### 10.6.5 示例5：损坏的Ack处理

**说明**：
1. Device A发送TLP 4094、4095、0、1、2
2. Device B成功接收TLP 4094、4095和0，NEXT_RCV_SEQ设置为1，返回Ack 0
3. Ack 0在链路上传输时有一位出错，Device A检查16位CRC时失败，丢弃Ack。这意味着TLP 4094、4095和0保留在Device A的重放缓冲区中
4. TLP 1和2到达Device B并良好，因此NEXT_RCV_SEQ计数递增到3，AckNak_LATENCY_TIMER再次到期后返回Ack 2
5. Ack 2安全到达Device A，清除重放缓冲区中的TLP 4094、4095、0、1和2

**如果Ack 2也丢失或损坏**：
如果Ack 2也丢失或损坏，且没有进一步的Ack或Nak DLLP返回给Device A，其REPLAY_TIMER将到期，导致重放其整个缓冲区。Device B看到TLP 4094、4095、0、1和2，认为它们是重复的（它们的序列号早于NEXT_RCV_SEQ计数3）。它们被丢弃，并返回另一个Ack 2。

[原文: 第346-347页]

### 10.6.6 示例6：损坏的Nak处理

**说明**：
1. Device A发送TLP 4094、4095、0、1、2
2. Device B成功接收TLP 4094、4095和0（AckNak_LATENCY_TIMER尚未到期）。下一个接收的TLP未通过LCRC检查，因此Device B设置NAK_SCHEDULED标志，重置并保持AckNak_LATENCY_TIMER。Nak返回，序列号为最后收到的良好TLP，0
3. Nak 0在Device A未通过16位CRC检查并被丢弃
4. 此时，Device B在成功收到下一个预期的TLP（本例中为TLP 1）之前不会发送任何Ack或Nak。然而，这需要重放。Device A还不知道需要重放，因为发送回的Nak已损坏并被丢弃
5. 这由REPLAY_TIMER解决。REPLAY_TIMER最终会到期，因为它在指定时间范围内没有看到表示前进进度的Ack或Nak
6. REPLAY_TIMER到期后，Device A将重放缓冲区中的所有TLP，递增REPLAY_NUM计数器，重置并重新启动REPLAY_TIMER
7. Device B将接收TLP 4094、4095和0，并识别它们是重复的。重复TLP将被丢弃，并调度Ack，序列号0
8. 一旦Device B成功收到TLP 1，它将清除NAK_SCHEDULED标志，递增NEXT_RCV_SEQ，并重新启动AckNak_LATENCY_TIMER

## 10.7 定时器值

### 10.7.1 REPLAY_TIMER方程

超时值主要取决于最大数据有效载荷和链路宽度。计算REPLAY_TIMER值的方程（以符号时间为单位）如下：

```
REPLAY_TIMER = (((Max_Payload_Size + TLPOverhead) * AckFactor) / LinkWidth + InternalDelay) * 3 + Rx_L0s_Adjustment
```

注意：该值是Ack/Nak延迟值的三倍。

**方程字段定义**：

| 字段 | 说明 |
|------|------|
| Max_Payload_Size | 设备控制寄存器中的值。如果有多个功能具有不同的Max_Payload_Size值，规范建议使用其中最小的一个 |
| TLP Overhead | TLP字段中超出数据有效载荷的额外部分（序列号、头部、摘要、LCRC和起始/结束帧符号）。在规范中，开销值被视为28个符号的常量 |
| AckFactor (AF) | 基本是一个调整因子，表示在必须发送Ack之前可以接收的最大有效载荷大小的TLP数量。AF值范围为1.0到3.0，旨在平衡链路带宽效率和重放缓冲区大小 |
| LinkWidth | 范围为x1（1位宽）到x32（32位宽） |
| InternalDelay | 在接收方处理TLP和在发送方处理DLLP（Ack）的内部延迟。该值在规范中以符号时间定义，取决于链路速度：Gen1 = 19，Gen2 = 70，Gen3 = 115 |
| Rx_L0s_Adjustment | 该值包含在1.x PCIe规范中，但在2.0及更高版本的PCIe规范中被删除。它可以用于考虑接收电路从L0s退出到L0所需的时间 |

## 10.8 交换机直通模式（Switch Cut-Through Mode）

### 10.8.1 背景

考虑一个示例，其中一个大TLP需要通过交换机，如下图所示。由于入口交换机端口在收到整个TLP之前无法判断数据包中是否有错误，因此它通常会存储整个数据包并在转发到出口端口之前检查错误。这种存储转发方法有效，但对于大数据包，通过交换机的延迟可能很大，这可能对某些应用造成问题。

### 10.8.2 延迟改进选项

由于TLP的第一部分包含数据包的路由信息头部，一个选项是假设数据包是良好的数据包，甚至在收到整个数据包之前就开始评估头部中的路由信息。这将允许交换机在评估路由后尽快开始将TLP转发到出口端口。

当然，入口端口在收到TLP末尾的LCRC之前无法检查数据包中的错误，因此涉及一些风险，即正在转发的数据包实际上可能包含错误。

### 10.8.3 直通操作

直通模式提供了前面描述的转发问题的解决方案：如果看到传入数据包中的错误，则必须"取消"已经在传输途中的数据包。

**取消的数据包（Nullified Packet）**：
- 以EDB（结束错误）符号而不是END（结束良好）符号终止
- TLP的32位LCRC从原始计算值反转（1的补码）
- 本质上，取消的数据包被视为从未存在过

**在交换机出口端口**：
- 重放缓冲区丢弃数据包
- NEXT_TRANSMIT_SEQ计数器减1（回滚）

**当设备收到取消的TLP时**：
- 简单地丢弃数据包，视为从未存在过
- NEXT_RCV_SEQ不递增
- AckNak_LATENCY_TIMER不启动
- NAK_SCHEDULED不设置
- 静默丢弃取消的TLP，不返回Ack/Nak

## 10.9 错误处理总结

### 10.9.1 Ack/Nak处理的错误情况

Ack/Nak协议保证TLP的可靠交付，尽管可能发生几种错误。下面的列表包括用于解决它们的纠正机制。

| 错误类型 | 解决方案 |
|----------|----------|
| TLP中的LCRC错误 | 接收方检测到LCRC错误并调度包含NEXT_RCV_SEQ计数-1的Nak。作为响应，发送方至少重放一个TLP，从失败的TLP开始 |
| 到达接收方数据链路层的TLP丢失（例如物理层检测到数据包问题并丢弃它） | 接收方检查所有接收TLP的序列号，期望它们按下一个顺序序列号到达。如果TLP丢失，下一个成功的序列号将无序。作为响应，接收方调度Nak，NRS计数-1，发送方至少重放一个TLP，从丢失的TLP开始 |
| 到达发送方的Ack或Nak损坏 | 发送方检测到DLLP中的CRC错误，丢弃数据包，等待下一个 |
| - Ack情况 | 后续Ack以较晚的序列号接收，导致发送方重放缓冲区清除序列号等于或早于它的所有TLP。发送方不知道有任何问题（除了重放缓冲区暂时填满的潜在情况） |
| - Nak情况 | 接收方已设置Nak调度标志，在成功收到下一个预期TLP之前不会发送另一个Nak或任何Ack，意味着需要重放。当然，如果Nak丢失，发送方不知道需要重放。在这种情况下，REPLAY_TIMER最终会到期并触发重放 |
| 在预期时间内未看到Ack/Nak | REPLAY_TIMER超时触发重放 |
| 接收方未能为接收的TLP发送Ack/Nak | 发送方的REPLAY_TIMER将到期并导致重放 |

[原文: 第349-350页]

### 10.9.2 数据包调度优先级

设备可能在给定链路上有许多类型的TLP、DLLP和有序集要传输。调度数据包的推荐优先级为：

1. 完成当前正在进行的任何TLP或DLLP（最高优先级）
2. 有序集（Ordered Set）
3. Nak
4. Ack
5. 流量控制（Flow Control）
6. 重放缓冲区重传
7. 在事务层等待的TLP
8. 所有其他DLLP传输（最低优先级）

## 10.10 Linux内核实现

在Linux内核中，PCIe数据链路层的Ack/Nak机制主要由硬件实现，内核通过AER（Advanced Error Reporting）机制与数据链路层交互。

### 10.10.1 AER中的数据链路层错误

**文件**：`drivers/pci/pcie/aer.c`

内核通过AER机制监控以下数据链路层错误：

| 错误类型 | 寄存器位 | 说明 |
|----------|----------|------|
| Bad DLLP | PCI_ERR_COR_BAD_DLLP (bit 7) | 检测到CRC错误的DLLP |
| Replay Rollover | PCI_ERR_COR_REP_ROLL (bit 8) | 重放计数器超过阈值 |
| Replay Timeout | PCI_ERR_COR_REP_TIMER (bit 12) | REPLAY_TIMER超时 |
| Data Link Protocol | PCI_ERR_UNC_DLP (bit 4) | 不可纠正的数据链路层协议错误 |

```c
// AER数据链路层错误掩码定义
#define AER_DATA_LINK_LAYER_ERROR_MASK(t) ((t == AER_CORRECTABLE) ? \
    (PCI_ERR_COR_BAD_TLP|                    \
    PCI_ERR_COR_BAD_DLLP|                    \
    PCI_ERR_COR_REP_ROLL|                    \
    PCI_ERR_COR_REP_TIMER) : PCI_ERR_UNC_DLP)

1. 错误检测（AER中断）
   ↓
2. 广播error_detected消息到设备驱动
   ↓
3. 如果是冻结状态，执行slot_reset
   ↓
4. 广播mmio_enabled消息
   ↓
5. 广播slot_reset消息（如需要）
   ↓
6. 广播resume消息
   ↓
7. 恢复完成或报告失败
```

**关键函数**：
```c
pci_ers_result_t pcie_do_recovery(struct pci_dev *dev,
        pci_channel_state_t state,
        pci_ers_result_t (*reset_subordinates)(struct pci_dev *pdev));
```

### 10.10.4 与硬件的交互

Linux内核不直接控制Ack/Nak协议（这是硬件自动完成的），但内核可以：

1. **监控错误**：通过AER寄存器监控数据链路层错误计数
2. **配置策略**：通过ECRC（End-to-End CRC）策略配置数据完整性检查
3. **错误恢复**：当检测到不可纠正错误时，执行设备恢复流程
4. **日志记录**：记录错误事件用于故障诊断

### 10.10.5 调试信息

内核提供以下接口查看数据链路层状态：

```bash
# 查看AER错误计数
$ cat /sys/bus/pci/devices/<device>/aer_dev_stats

# 查看PCIe链路状态
$ lspci -vv -s <device>
```

## 10.11 本章总结

Ack/Nak协议是PCIe数据链路层的核心机制，确保TLP在链路上的可靠传输。主要要点包括：

1. **协议目的**：提供自动的、基于硬件的错误检测和恢复机制

2. **关键组件**：
   - 发送方：NEXT_TRANSMIT_SEQ计数器、LCRC生成器、重放缓冲区、REPLAY_TIMER、REPLAY_NUM计数器
   - 接收方：NEXT_RCV_SEQ计数器、LCRC检查器、NAK_SCHEDULED标志、AckNak_LATENCY_TIMER

3. **基本流程**：
   - 发送方为每个TLP分配序列号并计算LCRC
   - 接收方验证LCRC和序列号
   - 成功接收返回Ack，失败返回Nak
   - 发送方根据Ack/Nak重放或清除TLP

4. **错误处理**：
   - LCRC错误：接收方发送Nak，发送方重放
   - TLP丢失：序列号检查检测，发送Nak，重放
   - Ack/Nak丢失：REPLAY_TIMER超时触发重放

5. **优化机制**：
   - Ack合并：一个Ack确认多个TLP
   - 直通模式：交换机提前转发减少延迟

6. **Linux实现**：
   - 硬件自动处理Ack/Nak协议
   - 内核通过AER监控错误
   - 提供错误恢复框架
